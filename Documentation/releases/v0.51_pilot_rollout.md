> Date: July 13, 2026

---

## 1. Pilot Overview

**Release:** v0.51 — Fireflies Automatic Interview Sync
**Goal:** Remove manual interview upload as the primary workflow for supported users.
**Target window:** two-week pilot, starting on rollout to the first connected pilot account

### 1.1 Pilot Objectives

1. Validate Fireflies connection and credential validation succeeds/fails clearly for real users
2. Confirm bounded backfill (`backfill_days`, default 90) materializes interviews correctly and idempotently
3. Confirm hourly `scheduled_connector_sync` keeps the interview library current without duplication
4. Verify disconnect and imported-data delete behavior matches trust/privacy copy
5. Gather first-use timing data (connection → first materialized interview) for onboarding tuning
6. Learn whether pilot users actually prefer auto-sync over manual upload

### 1.2 Pilot Candidate Criteria

| Criterion | Requirement |
|---|---|
| Active Fireflies account | Required, with API key access |
| Meeting history | At least 5 historical meetings within the 90-day backfill window |
| Team size | 1–3 users per pilot account |
| Engagement commitment | Weekly check-in during pilot window |

### 1.3 Target Pilot Accounts

| Account | Notes |
|---|---|
| Internal (Spec10x) | Dogfood — exercise connect, backfill, incremental sync, disconnect, delete first |
| Pilot A | Existing v0.5 pilot account with regular Fireflies-recorded interviews |
| Pilot B | New pilot account onboarded directly onto auto-sync (no manual upload history) |

## 2. Rollout Sequence

### Phase 1: Internal Validation

- [ ] Connect internal Fireflies workspace, confirm `validate()` succeeds
- [ ] Run backfill, confirm interviews materialize with full pipeline (insights, themes, chunks, Ask)
- [ ] Re-run backfill, confirm idempotency (no duplicate interviews or signals)
- [ ] Wait for one hourly `scheduled_connector_sync` cycle, confirm incremental sync picks up new meetings
- [ ] Delete a manually-materialized interview, confirm it is not recreated by the next sync (`test_user_deleted_interview_is_not_recreated` behavior holds in staging)
- [ ] Run disconnect, confirm `secret_ref` is cleared and sync stops
- [ ] Run imported-data delete, confirm materialized interviews/signals/source items are removed
- [ ] Confirm `/trust` and `/privacy` Fireflies rows match actual behavior

### Phase 2: Canary

- [ ] Enable Fireflies connector for internal account only in production
- [ ] Monitor `sync_runs` and structured logs for 24 hours
- [ ] Confirm no regression to existing manual-upload interview flow

### Phase 3: Pilot Rollout

- [ ] Enable for Pilot A and Pilot B
- [ ] Walk each pilot account through connect → backfill
- [ ] Monitor daily: sync success rate, backfill duration, duplicate rate, time-to-first-insight
- [ ] Collect feedback at end of Week 1 and Week 2, focused on: setup friction, trust in auto-sync, and manual-upload-vs-auto-sync preference

## 4. Rollback Plan

Same mechanism as `v0.5` (see `v0.5_pilot_rollout.md` §4): disable the connector for affected accounts, pause active `sync_runs`, roll back deployment if needed, notify pilot users. Fireflies-specific addition: on rollback, materialized interviews remain in place (they behave as native interviews) — only future sync is halted, not existing data.
