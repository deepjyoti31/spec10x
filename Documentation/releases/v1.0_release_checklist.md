## 1. Release Summary

**Release:** v1.0 — Full Loop MVP
**Goal:** Close the loop: an approved, evidence-cited brief breaks itself into agent-ready tasks, hands off as one self-contained markdown bundle, and reports back on whether the pain it addressed went quiet after shipping.

Delivered:

- **Task breakdown engine** (`EPIC-10-01`): `tasks_json` / `tasks_generated_at` / `shipped_at` on `specs` (migration `d2e7f4a9c6b3`), Gemini breakdown service fed only by the spec's own sections and evidence snapshot, 12-task cap, S/M/L normalization, dependency and citation sanitizing; `POST /api/specs/{id}/tasks` gated on approval (`D-10-02`), failures persist nothing (`D-10-03`)
- **Agent handoff** (`EPIC-10-02`): `GET /api/specs/{id}/export` — self-contained markdown bundle (brief + tasks + numbered evidence appendix, `D-10-04`); Spec Studio task panel with `Break into tasks`, complexity/dependency/citation chips, `Copy for agent` and `.md` download; `/tasks` delivery workbench
- **Ship & outcomes loop** (`EPIC-10-03`): first transition to Shipped stamps `shipped_at` (kept across rollback, `D-10-06`); `GET /api/specs/outcomes` — 4 weekly voice buckets each side of ship with `improving` / `worsening` / `flat` / `too_early` / `unavailable` states; `/outcomes` page with validated pre/post bars and correlational framing
- **Roadmap & Command Center** (`EPIC-10-04`): `/roadmap` Now/Next/Later/Shipped derived purely from spec statuses (`D-10-05`); `spec_pipeline` on the home dashboard replacing both "Coming Soon" placeholders; `Tasks`, `Roadmap`, `Outcomes` sidebar entries unlocked — every section of the v1.0 shell is now live

Explicitly deferred per `PRD-10-01` (below the cut line, not dropped): two-way PM tool sync (Linear/Jira/GitHub/Asana), public API & webhooks, competitive intelligence, custom dashboards/reports, auto-close loop notifications, task editing/reordering.

## 3. Rollout Notes

- No feature flag — `/tasks`, `/roadmap`, `/outcomes` are additive surfaces. Changes to existing surfaces: Spec Studio gains the task panel and export buttons, the home page's two "Coming Soon" placeholders become the live Spec Pipeline, and the Deliver/Learn sidebar items unlock.
- Task breakdown requires the same Vertex AI access as spec generation — no new infrastructure.
- Breakdown runs synchronously in-request like spec generation; it is one bounded call on an already-generated document.
- Rollback is a plain deployment rollback plus `alembic downgrade -1` — the three new columns are nullable and nothing in v0.8 behavior depends on them.
- Operational reminder (carried from v0.8): full-suite test runs should use a fresh database via `DATABASE_URL`/`DATABASE_URL_SYNC` override; the long-lived dev database makes signal-consistency tests orders of magnitude slower (16 v0.8 tests: 17 minutes on the dev DB vs. the same suite in seconds on a fresh one).

## 4. Measurement Plan (to schedule with pilot rollout)

| Metric | Why It Matters | Source Of Truth | Decision Use |
|---|---|---|---|
| percent of approved specs broken into tasks | is the breakdown step actually used | `tasks_generated_at` non-null vs. approved specs | if skipped, the handoff starts elsewhere — improve the panel before building PM sync |
| export copies/downloads per approved spec | does the bundle reach a coding agent | product analytics on the export actions | low usage + high task generation → the bundle format is wrong, ask pilots what their agent needs |
| task regeneration rate | breakdown quality proxy | repeat `POST /tasks` per spec | heavy regeneration → prompt/granularity work before task editing features |
| percent of shipped specs with a non-`too_early` outcome read after 4 weeks | does the loop actually close | `/outcomes` states | many `unavailable` → theme lifecycle problem; many `flat` → outcome window or measure needs rethink |
| roadmap + outcomes weekly usage | do the derived views earn their place | product analytics | if weak, fold them into board/home rather than expanding reporting |

## 5. What's Next

- Run the v0.8 + v1.0 measurement plans together through the pilot window. The loop is now feature-complete end to end; everything beyond (PM tool sync, API/webhooks, wireframes per `D-08-04`) should be pulled by pilot demand, not pushed.
- Fast-follow candidates, strictly after measurement: Playwright smoke paths for specs/tasks/outcomes, task export to GitHub Issues if pilots ask, spec collaboration once multi-user workspaces open.
