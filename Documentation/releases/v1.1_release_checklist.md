## 1. Release Summary

**Release:** v1.1 — Open-Source Readiness, Multi-User Workspaces, and Delivery Integrations
**Goal:** Close the adoption gaps the open-source pivot exposed: strangers can run it in one command with CI behind every PR, teams can share a workspace, coding agents can pull specs and report ships without copy-paste, tasks reach GitHub, and the Learn column pushes instead of waiting to be read.

Delivered:

- **Open-source onboarding & CI** (`EPIC-11-01`): docker compose `app` profile (one-shot `migrate`, `backend`, `worker`, `frontend`; infra-only default preserved); `.github/workflows/ci.yml` running the backend suite against a fresh `pgvector/pgvector:pg17` + Redis + MinIO (D-11-06), frontend lint + production build, and the Playwright smoke suite; `CONTRIBUTING.md`; README rewritten quickstart-first with the "AI features need Vertex AI" matrix
- **Multi-user workspaces** (`EPIC-11-02`): `workspace_members` + `users.active_workspace_id` (migration `a4b8c2d9e1f7`, clean downgrade); workspace API (invite by email, explicit accept per D-11-02, revoke/remove/leave with instant personal fallback, switch); `get_scoped_user` request scoping so every data surface shares the workspace owner's pool (D-11-01) — 12 API modules swapped, identity routes untouched, websocket subscribes to the scoped pool; `/team` wired to real data with the seat/plan mockup framing removed
- **MCP server** (`EPIC-11-03`): stdio server (`python -m app.mcp`, `SPEC10X_MCP_USER_EMAIL` identity) with `list_specs`, `get_spec_bundle` (byte-identical to `GET /api/specs/{id}/export`), `get_spec_outcome`, and `mark_spec_shipped` (the only mutation, D-11-04, reusing the D-10-06 first-ship rule); logic lives in SDK-free `app/mcp/tools.py`
- **Full-loop smoke** (`EPIC-11-04`): `frontend/e2e/smoke/specs-loop.spec.ts` — 5 tests covering `/specs` → Spec Studio (sections, evidence, tasks) → GitHub export modal → `/tasks` → `/outcomes` with the correlational caution asserted
- **Delivery integrations** (`EPIC-11-05`): `POST /api/specs/{id}/tasks/github` — one issue per task via request-scoped PAT (D-11-03: never persisted, logged, or echoed back), per-task idempotency with issue links saved into `tasks_json`, modal + issue chips in Spec Studio and `/tasks`; outcomes computation extracted to `app/services/outcomes.py` and shared with the daily `scheduled_outcome_notifications` cron that notifies each spec owner exactly once when the readout first leaves `too_early` (D-11-05)

Explicitly deferred per `PRD-11-01` (below the cut line, not dropped): two-way PM tool sync, public REST API/webhooks, roles beyond owner/member, workspace audit logs, multiple owned workspaces, email delivery of invites/notifications, per-member attribution inside shared data.

## 3. Rollout Notes

- All new surfaces are additive: `/team` goes from static mockup to real, Spec Studio and `/tasks` gain the GitHub export affordance, task cards gain issue-link chips. No existing behavior changes for a solo user with no members — `get_scoped_user` resolves to the requester themselves unless they explicitly joined a shared workspace.
- The migration adds one table and three nullable columns; rollback is a plain deployment rollback plus `alembic downgrade -1`.
- The MCP server is opt-in tooling run from the operator's own environment; it ships with the backend but starts nothing by default.
- The outcome-notification cron runs daily at 06:30 in the worker; it is idempotent and safe to backfill (first run notifies owners of any already-readable shipped specs exactly once).
- Compose default behavior is unchanged (`docker compose up` = infra only); the full stack requires the explicit `--profile app`.
- New dependency: `mcp==1.28.1` with `starlette==0.46.2` pinned — do not bump starlette independently of fastapi.

## 4. Measurement Notes
