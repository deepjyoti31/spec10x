# Spec10x v0.5 Sync Observability Runbook

> Date: March 23, 2026
> Status: Sprint 6 baseline implemented
> Audience: Founder and operators
> Scope: `v0.5` pilot review surface for sync health and first-use value

## 1. Purpose

This runbook defines the review surface for Sprint 6 observability.

It is intentionally not a new in-product admin UI.

The review surface for `v0.5` is:

- `sync_runs` as the durable audit source of truth
- structured sync logs emitted by `backend/app/services/sync_orchestrator.py`
- one saved dashboard or query view in the logging tool used during pilot review

## 2. Structured Log Events

The orchestrator emits these event names:

- `sync_run_started`
- `sync_run_succeeded`
- `sync_run_failed`

Each event should carry the following fields:

- `provider`
- `source_type`
- `workspace_id`
- `connection_id`
- `sync_run_id`
- `run_type`
- `status`
- `records_seen`
- `records_created`
- `records_updated`
- `records_unchanged`
- `duplicate_rate`
- `duration_ms`
- `time_to_first_insight_ms`
- `error_summary` when present

## 3. Metric Definitions

Use these definitions consistently:

- `sync_success_rate`: succeeded runs divided by total runs for the selected window
- `sync_failure_rate`: failed runs divided by total runs for the selected window
- `duplicate_rate`: `records_unchanged / records_seen`, rounded to four decimals
- `time_to_first_insight_ms`: first successful connection or import run that creates useful evidence; current baseline uses the first successful run duration when new evidence is created on first load
- `last_successful_sync`: latest `finished_at` for a succeeded run on an active connection

## 4. Saved Review Surface

The founder review surface should answer five questions:

1. Which providers are failing right now?
2. Which active connections have gone stale?
3. Which syncs are producing high duplicate rates?
4. How long do backfills take?
5. How long does it take from connect or import start to first useful insight?

Minimum saved views:

- Provider health table grouped by `provider`, showing total runs, success rate, failure rate, and average `duration_ms`
- Active connection table grouped by `connection_id`, showing latest status, latest success time, latest failure time, and last `duplicate_rate`
- First-use table filtered to first successful runs with non-null `time_to_first_insight_ms`
- Failure log view filtered to `event = sync_run_failed`

## 5. Alert Thresholds

Use these thresholds during `v0.5` pilot review:

| Signal | Threshold | Why it matters | First response |
|---|---|---|---|
| consecutive failures | 3 failed runs on the same active connection | likely broken credentials or provider drift | inspect last three failures, validate credentials, retry manually |
| stale connection | no successful sync in 24 hours for an active connection | product is silently going stale | inspect last run, trigger manual sync, confirm provider availability |
| duplicate rate spike | `duplicate_rate > 0.02` on a run | retry or cursor logic may be drifting | inspect cursor, compare `records_seen` to `records_unchanged`, replay fixture if needed |
| slow backfill | `duration_ms > 1800000` on a pilot backfill | first-use value is too slow | reduce history window or improve batching before wider rollout |
| slow first insight | `time_to_first_insight_ms > 3600000` | setup feels broken to pilots | inspect first-run path and evidence creation timing |

## 6. Review Cadence

During the `v0.5` pilot window:

- review provider health daily
- review stale connections daily
- review failure logs after every failed pilot sync report
- review first-use timing at least once per pilot onboarding
- summarize trends weekly before the `G1M` measurement decision

## 7. Escalation Notes

Use this order of operations when a threshold trips:

1. confirm the failing `connection_id`, `provider`, and `sync_run_id`
2. read the latest structured log entry and matching `sync_runs` row
3. determine whether the issue is credentials, provider change, rate limit, cursor drift, or duplicate handling
4. run the smallest safe retry possible
5. if duplicate behavior or provider drift is confirmed, block broader rollout work until the connector path is stable
