## 1. Pilot Overview

**Release:** v0.5 — Multi-Source Evidence Alpha
**Target window:** May 12–23, 2026 (two-week pilot)
**Ship gate (G1):** May 10, 2026 — all committed Sprint 7 stories accepted, smoke tests green

1. Validate Zendesk backfill and incremental sync reliability with real customer data at pilot scale

## 4. Rollback Plan

### 4.1 Trigger Conditions

Initiate rollback if any of the following occur during pilot:

- **3+ consecutive sync failures** on any pilot connection without resolution within 4 hours
- **Data integrity issue** — duplicate signals, missing evidence, or corrupt normalization
- **Security concern** — credential exposure, unauthorized data access
- **Pilot-blocking UX bug** — user cannot complete core flow (connect → sync → view feed → triage board)

### 4.2 Service Rollback

| Step | Action | Owner | Duration |
|---|---|---|---|
| 1 | Disable `v0.5_pilot_enabled` feature flag for affected accounts | Ops | < 5 min |
| 2 | Pause all active sync runs (`SyncRunStatus.running`) for affected connections | Ops | < 10 min |
| 3 | Roll back backend deployment to last known-good revision | Eng | < 15 min |
| 4 | Roll back frontend deployment to last known-good revision | Eng | < 15 min |
| 5 | Notify affected pilot users with status update and ETA | PM | < 30 min |

### 4.3 Data Rollback

| Concern | Action |
|---|---|
| Corrupted signals | Soft-delete signals created after the corruption timestamp for affected `workspace_id` |
| Duplicate source items | De-duplicate via `uq_source_items_connection_external_id` constraint — rerun sync after fix |
| Orphaned sync runs | Mark as `failed` with error summary pointing to rollback |

### 4.4 Communication Template

```
Subject: Spec10x v0.5 Pilot — Service Update

Hi [Pilot Contact],

We've identified an issue with [brief description] during the v0.5 pilot.
As a precaution, we've temporarily disabled the multi-source features for
your account while we resolve this.

Your existing interview data and analysis are unaffected.

We expect to re-enable the pilot within [ETA]. We'll follow up with a
detailed post-incident summary.

— Spec10x Team
```

## 5. Post-Pilot

### 5.1 Success Metrics

| Metric | Target | Measurement |
|---|---|---|
| Sync success rate | ≥ 98% | `sync_runs` table |
| Duplicate rate | ≤ 2% | `records_unchanged / records_seen` |
| Time to first insight | ≤ 60 min | First successful sync with `records_created > 0` |
| Pilot user satisfaction | Qualitative positive | Feedback interviews |
| Zero data integrity incidents | 0 | Incident log |

### 5.2 G1M Measurement Gate

After the pilot window closes, review all success metrics and pilot feedback.
The G1M gate determines whether v0.5 proceeds to general availability or
requires additional hardening sprints.

| Decision | Criteria |
|---|---|
| **Proceed to GA** | All success metrics met, no unresolved P0/P1 defects |
| **Extend pilot** | One or two metrics missed with clear fix path |
| **Revert to v0.4** | Fundamental reliability or trust issue requiring architectural changes |
