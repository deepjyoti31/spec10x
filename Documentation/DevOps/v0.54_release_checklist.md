## 1. Release Summary

**Release:** v0.54 — Deferred Backlog Cleanup
**Goal:** Clear cut-line debt from `v0.5`–`v0.53` before opening any new release scope (`v0.8` groundwork or a fourth connector).

Delivered:

- **Saved Views** (`EPIC-054-01`): `SavedView` model, CRUD API, `Views` dropdown on `/feed` to save/apply/delete named filter sets
- **Collections & Theme Merge** (`EPIC-054-02`): `Collection` model with interview membership, new `/collections` pages, `POST /api/themes/{id}/merge` with a `Merge` control on `/board`
- **Ownership Polish** (`EPIC-054-03`): nullable `workspace_id` on `Interview`/`Theme` backfilled from each owner's personal workspace, plus a private `comment` field on both, surfaced via `PATCH` on interviews and themes and a note card on the interview detail page

No multi-user UI shipped — `EPIC-053-03`'s full ownership-model revisit (invites, roles, workspace admin UX) remains forecast-only.

### 2.2 Data Integrity Notes

| # | Gate | Status | Evidence |
|---|---|---|---|
| D1 | Theme merge never drops evidence | ✅ | Merge reassigns via ORM relationship assignment (`insight.theme = target_theme`) rather than raw FK writes — this was a real bug caught during implementation: setting `insight.theme_id` directly left the stale `source_theme.insights` collection in the identity map, and SQLAlchemy's delete-time FK nullification silently reset the reassigned rows back to `NULL`. Fixed by using relationship assignment, which correctly re-parents insights and sub-themes and empties the source collection before delete. Covered by `test_merge_reassigns_insights_and_sub_themes` |
| D2 | `workspace_id` backfill is deterministic | ✅ | Migration backfill joins strictly on `owner_user_id = <table>.user_id AND kind = 'personal'`, guaranteed to exist per user via `get_or_create_default_workspace` |
| D3 | Collection membership add is idempotent | ✅ | `test_add_and_remove_interviews` — re-adding an already-member interview is a no-op, not an error |

## 3. Rollout Notes

- No feature flag — all three features are additive UI surfaces (a dropdown on `/feed`, a new `/collections` section, a button on `/board` cards) with no change to existing user-visible behavior when unused.
- No pilot measurement window — this release clears debt rather than testing a new hypothesis, so it does not get its own two-week measurement plan. It rolls out alongside whatever `v0.51`/`v0.52`/`v0.53` pilot rollout is already in progress.
- Rollback, if needed, is a plain deployment rollback plus `alembic downgrade -1` — the migration's downgrade path drops the new tables/columns cleanly since nothing in the existing `v0.5x` product depends on them.
