## 1. Release Summary

**Release:** v0.8 — Specification Engine MVP
**Goal:** Turn a prioritized theme into a complete, evidence-cited feature brief in one action, and unlock the Trends surface — the bridge from "know what to build" to "ready to build it."

Delivered:

- **Spec generation** (`EPIC-08-01`): `Spec` model (workspace-owned, theme FK `SET NULL`, evidence snapshot), Gemini generation service with a numbered-evidence citation contract, bounded input (40 voice signals + 5 metric windows per call), citation sanitizing
- **Specs API & review workflow** (`EPIC-08-02`): `POST/GET/PATCH/DELETE /api/specs` + `/regenerate`; Draft → In Review → Needs Changes → Approved → In Dev → Shipped with server-validated transitions; regeneration blocked after approval
- **Spec Studio** (`EPIC-08-03`): `/specs` list with status tabs and theme-picker generator; `/specs/[id]` split-pane studio (markdown sections with inline edit, citation chips that focus the evidence panel, status controls, regenerate/delete); live `Generate Spec` on `/board` pinned cards; `Specs` and `Trends` unlocked in the sidebar
- **Trends** (`EPIC-08-04`): `GET /api/themes/trends` (8 rolling weekly voice-signal buckets, direction from the existing `calculate_theme_trend`) and the `/trends` page (validated series palette, crosshair tooltip, table view, Rising/Declining/Stable sections)

Explicitly deferred per `D-08-04` (below the cut line, not dropped): wireframe suggestions, user flow diagrams, codebase awareness (repo connectors), spec collaboration (comments, co-editing, sign-offs), task export.

## 3. Rollout Notes

- No feature flag — `/specs` and `/trends` are additive surfaces. The only change to an existing surface is the board's `Generate Spec` button going from a disabled "SOON" placeholder to live.
- Spec generation requires valid Vertex AI access in the deployed environment (same dependency as interview analysis and Ask — no new infrastructure).
- Generation runs synchronously in-request (`D-08-02`); typical latency is one Gemini structured-output call. If briefs grow or latency becomes a complaint, move generation to the arq worker as a follow-up.
- Rollback is a plain deployment rollback plus `alembic downgrade -1` — nothing in the existing v0.5x product depends on the `specs` table.

## 4. Measurement Plan (to schedule with pilot rollout)

| Metric | Why It Matters | Source Of Truth | Decision Use |
|---|---|---|---|
| specs generated per active pilot user | is one-action brief generation actually used | `specs` table counts | if unused, revisit entry points before expanding the engine |
| percent of specs edited after generation | AI-draft quality proxy — some editing is healthy, wholesale rewrites are not | `is_edited` + section diffs in pilot review | heavy rewrites → improve prompt/evidence quality before wireframes |
| percent of specs reaching Approved | does the workflow reflect real adoption | `specs.status` distribution | if specs stall in Draft, the brief is not trusted — hold `v0.8x` expansion |
| citation click-through in Spec Studio | do users verify evidence (trust behavior) | product analytics | low verification + low trust feedback → make citations more prominent |
| trends page weekly usage | does velocity framing earn a place in the workflow | product analytics | if weak, fold trends into board/insights rather than expanding it |
