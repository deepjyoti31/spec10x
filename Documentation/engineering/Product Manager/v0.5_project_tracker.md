## 1. Purpose

This document is the operational tracker for the `v0.5x` release family:

- `v0.5`
- `v0.51`
- `v0.52`
- optional `v0.53`

It now does four jobs, not one:

1. controls scope and cut lines
2. controls release gates and measurement windows
3. tracks the current release backlog at story level
4. holds later releases as forecast backlog only

Important rule:

- only `v0.5` is detailed enough to become sprint-committed today
- `v0.51+` remain forecast backlog until the prior release has shipped and been measured

That rule fixes the earlier contradiction between "do not start the next release until the current one is measured" and the old back-to-back calendar.

## 2. Operating Model

### 2.1 Delivery assumptions

- Sprint length: `1 week`
- Nominal capacity: `20 points`
- Hard committed capacity: `18 points`
- Reserve buffer: `2 points`
- Estimation scale: `1, 2, 3, 5, 8`
- Any story larger than `8` must be split before commitment

### 2.2 Planning levels and workflow states

Use these states consistently:

- `Outline` - forecast placeholder, not commitment-ready
- `Draft` - shaped enough to discuss, but still missing readiness inputs
- `Ready` - meets Definition of Ready and can enter the next sprint
- `In Sprint`
- `In Review`
- `Accepted`
- `Released`
- `Measured`

Rule:

- only `Ready` stories may enter a sprint
- all `v0.51+` stories stay in `Outline` until the previous release passes its measurement gate

### 2.3 Definition of Ready

A story is `Ready` only if all of the following are true:

- the user problem is clear
- the story maps to exactly one epic and one PRD
- acceptance criteria are testable
- dependencies and assumptions are listed
- the story fits inside one sprint
- acceptance evidence is pre-declared
- copy, trust, or observability needs are called out if the work is operationally sensitive

### 2.4 Definition of Done

A story is `Done` only if all of the following are true:

- implementation is complete
- acceptance criteria pass
- tests were added or updated where appropriate
- docs or UI copy were updated if behavior changed
- metrics or logs exist for operationally sensitive work
- required acceptance evidence is attached
- the maintainers have accepted the result

### 2.5 Required acceptance evidence

Every accepted story should link or point to all applicable evidence:

- PR or commit reference
- automated tests by name or test file
- screenshot, short demo clip, or before/after note for UI-visible work
- metrics or log sample for operational work
- documentation or trust-copy update note

### 2.6 Scope control rules

Stories are classified as:

- `Must` - required for the release ship decision
- `Should` - valuable, but cut before threatening release quality
- `Could` - only pull in if the release is green late in the cycle

Rules:

- only `Must` stories fill committed sprint points
- `Should` and `Could` items may be pulled only if the sprint is green by Thursday review
- no new `Must` stories after Sprint 2 without an explicit scope trade
- no new `Should` or `Could` stories after Sprint 5
- release freeze starts 48 hours before a ship gate; only defect fixes, copy fixes, and instrumentation fixes may merge

### 4.1 `signals` schema v1 decision

Decision summary:

- approve `signals` as an additive normalized evidence layer for `v0.5`
- do not replace `interviews`, `insights`, `themes`, or `transcript_chunks` in `v0.5`
- use `signals` as the shared query layer for unified feed, source filters, score inputs, and source-aware theme evidence
- use `workspace_id` as the primary ownership field for new source tables, while current user-scoped APIs resolve through a default personal workspace mapping

Why this is the right cut for `v0.5`:

- the current codebase already depends on `interviews`, `insights`, `themes`, and `transcript_chunks` for upload, synthesis, and Ask
- replacing that model now would create unnecessary rewrite risk
- adding `signals` lets multi-source work land without breaking the proven v0.1 interview path

Approved supporting table minimums:

| Table | Purpose | Minimum columns |
|---|---|---|
| `data_sources` | catalog of supported source types and providers | `id`, `source_type`, `provider`, `display_name`, `connection_method`, `is_active`, timestamps |
| `source_connections` | one workspace's connection or import pathway for one provider | `id`, `workspace_id`, `created_by_user_id`, `data_source_id`, `status`, `secret_ref`, `config_json`, `last_synced_at`, `last_error_summary`, timestamps |
| `sync_runs` | audit trail of backfills, incrementals, and manual re-syncs | `id`, `source_connection_id`, `run_type`, `status`, `started_at`, `finished_at`, `cursor_in`, `cursor_out`, `records_seen`, `records_created`, `records_updated`, `error_summary` |
| `source_items` | idempotent mapping from upstream records to internal records | `id`, `workspace_id`, `source_connection_id`, `external_id`, `source_record_type`, `external_updated_at`, `native_entity_type`, `native_entity_id`, `checksum`, `first_seen_at`, `last_seen_at`; unique on `source_connection_id + external_id` |
| `signals` | normalized evidence rows consumed by feed, score, and source-aware evidence views | `id`, `workspace_id`, `source_connection_id`, `source_item_id`, `source_type`, `provider`, `signal_kind`, `occurred_at`, `title`, `content_text`, `author_or_speaker`, `sentiment`, `source_url`, `metadata_json`, `native_entity_type`, `native_entity_id`, `status`, timestamps |

Approved `signals` v1 field rules:

- `source_type` is one of: `interview`, `support`, `survey`, `analytics`
- `provider` is provider-specific but not product-logic-specific, for example `native_upload`, `zendesk`, `csv_import`, `fireflies`, `mixpanel`
- `signal_kind` describes the normalized evidence unit, not the provider payload, for example `insight`, `ticket`, `survey_response`, `metric_window`
- `content_text` is the main evidence text used by feed rendering, scoring inputs, and later theme evidence views
- `metadata_json` holds provider-specific richness that should not leak into core product logic
- `native_entity_type` and `native_entity_id` link signals back to the current native data model so the UI can deep-link without rewriting v0.1 internals
- `status` should support at least `active`, `hidden`, and `error`

Approved source mapping rules:

- interview uploads map to one signal per accepted `insight`, with `provider = native_upload`, `native_entity_type = insight`, and `native_entity_id = insights.id`
- Zendesk maps to one signal per normalized ticket evidence item defined in `PRD-05-02`; the exact ticket-versus-comment rule belongs in the PRD, not in the core schema
- survey and NPS CSV imports map to one signal per response row, with `provider = csv_import`
- future analytics providers map to one signal per normalized event aggregate or metric window, not to raw event payload spam

Non-goals for `signals` v1:

- do not store raw secrets or auth tokens anywhere in source tables
- do not store giant full provider payloads in `signals`
- do not force Ask or transcript search onto `signals`; Ask remains grounded in `transcript_chunks`
- do not add Zendesk-only fields to core `signals` columns
- do not rewrite the existing interview-theme-insight system during `v0.5`

Decision consequence:

- `EPIC-05-01`, `EPIC-05-03`, `EPIC-05-04`, and `EPIC-05-05` may now design against this approved schema contract
- workspace-safe ownership details are closed by `D-05-03`

### 4.2 Ownership model decision

Decision summary:

- `data_sources` remains global and not customer-owned
- `source_connections`, `source_items`, and `signals` are workspace-owned from day one
- `sync_runs` inherit ownership through `source_connection_id`
- `source_connections` must store `created_by_user_id` for audit and support context
- `v0.5` can use a default personal workspace pattern behind the scenes, but no multi-user workspace UI ships in `v0.5`
- current user-scoped app surfaces should resolve the caller to their default workspace without changing the visible product model

Boundary rules:

- do not retrofit the existing `interviews`, `insights`, `themes`, or `transcript_chunks` tables to full workspace ownership during `v0.5`
- do not block source foundation work on invites, roles, collaboration, or workspace admin UX
- do ensure every new source table and index can survive a later move to true shared workspaces without another ownership rewrite

Decision consequence:

- `PRD-05-01` must include the default personal workspace bootstrap assumption
- new connector work should never assume a source connection is permanently personal-user-owned

### 4.3 Frontend smoke coverage decision

Decision summary:

- use Playwright as the `v0.5` frontend smoke test runner
- the `v0.5` ship blocker is smoke coverage, not a broader unit-test framework decision
- run Chromium-only in CI for `v0.5` to keep setup small and reliable

Approved suite shape:

- test location: `frontend/e2e/smoke/`
- target command shape: `npm run test:smoke`
- execution path: build app, start preview server, run Playwright smoke suite
- CI path: add a Cloud Build step after frontend build and before deploy

Minimum smoke coverage:

- integrations shell renders and loads provider state
- survey import happy path reaches preview and completion UI
- unified feed renders mixed-source rows
- source filters apply without breaking page state
- priority board renders ranked themes
- critical nav path between dashboard, feed or theme detail, and board is intact

Decision consequence:

- `US-05-06-01` should implement this exact baseline, not an open-ended frontend testing strategy

### 4.4 Sync observability baseline decision

Decision summary:

- `sync_runs` is the durable audit source of truth
- structured JSON logs go to Cloud Logging for operational debugging
- exceptional failures continue to surface through Sentry where appropriate
- maintainer review happens through one saved dashboard or query view, not through a new in-product admin UI in `v0.5`

Required metrics:

- sync success rate by provider
- sync failure rate by provider
- average backfill duration
- duplicate import rate
- time from connection or import start to first useful insight
- last successful sync per active connection
- records seen, created, updated, and unchanged per sync run

Initial alert thresholds:

- 3 consecutive failed syncs on the same active connection
- no successful sync in 24 hours for an active production connection
- duplicate import rate above 2 percent on a sync run
- backfill duration above 30 minutes for a pilot connection
- time-to-first-insight above 60 minutes after connection kickoff

Decision consequence:

- `US-05-06-04` should implement these metrics and thresholds as the minimum pilot review baseline

### 4.5 Trust package decision

Decision summary:

- `v0.5` ships with a customer-facing trust package, not just internal notes
- trust promises must stay plain-English and must match actual product behavior
- do not promise zero retention, provider-side deletion, or model-provider behavior that has not been verified in production settings

Approved trust package assets:

- one-page security and trust overview
- plain-English permissions summary by connector type
- storage and retention summary
- disconnect and imported-data delete guide
- subprocessor list
- privacy and terms language review checklist
- customer-facing trust promises document

Approved trust promises:

- Spec10x asks for read-only access where possible
- Spec10x stores the minimum durable copy needed to make the product useful
- Spec10x stores provider credentials in Secret Manager references, not ordinary app tables
- disconnect stops future syncs in Spec10x
- imported-data delete removes Spec10x's copied data, not upstream provider records
- Spec10x does not use customer content to train Spec10x's own generalized models
- Spec10x will not claim stronger model-provider retention guarantees than the production configuration can support

Decision consequence:

- `US-05-06-03` should ship against this trust package scope and promise set

## 16. Pull Request Strategy

Large releases fail when PRs become too large to review or reason about.

### 16.1 PR slicing rules

- one PR should ideally map to one story or one vertical slice of a story
- avoid mega PRs that combine migration, backend logic, UI, copy, and tests for unrelated work
- every PR should include its story ID in the title or description
- every PR should state what acceptance criteria it covers
- every PR for operational work should state what logs or metrics were added

Recommended PR size:

- small to medium
- reviewable in under 30 minutes

### 16.2 Evidence block template for PRs

Every PR description should include:

```text
Story: US-XX-YY-ZZ
PRD: PRD-XX-YY
Acceptance criteria covered:
- ...
- ...

Evidence:
- Tests:
- UI proof:
- Metrics/logs:
- Docs/copy:
```

## 17. Sprint Review Checklist

Every sprint review should answer:

- what stories were accepted?
- what slipped, and why?
- what new information changed priority or scope?
- what technical debt was created?
- what moved above or below the release cut line?
- what gate risk changed from green to yellow or red?

For connector-related sprints, also answer:

- did we request more permission than intended?
- can a failed sync be debugged from logs alone?
- can the user understand connection state without support?
- does disconnect and imported-data delete behavior still match the docs?

## 18. Release Go Or No-Go Checklist

Before shipping any release in the `v0.5x` family, confirm:

- release exit criteria are met
- all `Must` stories are accepted
- open defects meet the severity threshold
- key happy-path flows were tested
- key failure-path flows were tested
- docs and trust copy match actual behavior
- metrics and logs exist for new operational surfaces
- rollback or containment plan is understood
- release notes and pilot communication are ready

For connector releases, also confirm:

- disconnect behavior works
- imported-data delete behavior works
- secrets are not stored in ordinary DB fields
- provider-specific error states are understandable
- audit trail or sync trail is reviewable

Allowed ship decisions:

- `Go`
- `Hold`
- `Go with documented Sev-2 exceptions only`

## 20. Final Delivery Discipline

The tracker should be run with one core discipline:

- do not start the next release until the current release has both shipped and been measured

That matters because the whole `v0.5x` family is connected:

- `v0.5` proves Spec10x can combine evidence
- `v0.51` proves Spec10x can stay current automatically
- `v0.52` proves Spec10x can connect customer language to product behavior without losing trust
- `v0.53` is optional validation that the connector platform is reusable and supportable

If the team protects cut lines, keeps only the current release at `Ready` detail, and insists on gate evidence before moving forward, this tracker is realistic for a small maintainer team working with AI agents.
