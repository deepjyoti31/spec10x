# Spec10x v0.5 Trust Promises

> Date: March 16, 2026
> Status: Approved draft for `v0.5` implementation
> Audience: Customer-facing
> Owner: Founder acting as Product Manager

## What We Promise

### 1. Least-privilege access

Spec10x asks for read-only access where possible. We are an analysis product, not a writeback system, so `v0.5` should not require permissions to modify tickets, survey systems, or interview sources.

### 2. Minimum durable copy

Spec10x stores the minimum durable copy needed to make the product useful:

- transcript text where required for analysis
- normalized evidence rows
- embeddings
- source links and IDs
- sync cursors and sync status
- metadata needed for feed, theme evidence, score explanation, and support debugging

We do not position `v0.5` as "store everything forever."

### 3. Credentials are handled as secrets

Provider credentials should be stored through Secret Manager references, not ordinary application tables.

### 4. Clear disconnect behavior

Disconnect means Spec10x stops future syncs for that connection.

Disconnect does not mean upstream provider records are deleted.

### 5. Clear imported-data delete behavior

Imported-data delete means Spec10x removes its copied data for the requested connection or source scope.

Imported-data delete does not delete data in the upstream provider.

### 6. Honest model-data language

Spec10x does not use customer content to train Spec10x's own generalized models.

Spec10x should not claim stronger model-provider retention guarantees than the production configuration can actually support. If a provider has caching, logging, abuse monitoring, or retention caveats, those should be disclosed plainly.

### 7. Plain-English storage explanation

We explain:

- what we read
- what we store
- why we store it
- how disconnect works
- how imported-data delete works

### 8. Subprocessors are published

Spec10x should publish a current subprocessor list and keep it aligned with real infrastructure.

## What We Should Avoid Saying

- "We never store your data" if we keep a durable copy
- "Disconnect deletes your provider data" because it does not
- "Zero retention" unless production configuration and vendor terms support that claim
- "We do not send data to model providers" because model execution requires it

## v0.5 Trust Package Assets

- one-page security and trust overview
- permissions summary by source type
- storage and retention summary
- disconnect and imported-data delete guide
- subprocessor list
- privacy and terms review checklist
